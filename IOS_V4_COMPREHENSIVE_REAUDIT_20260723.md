## 3. 本轮已修复并通过静态检查

### 3.1 语言与底部导航

- 主应用默认模式由固定中文改为“跟随系统”。
- 中文系统语言统一解析为 `zh-Hans`。
- 英文及日语等非中文系统语言统一回退英文，避免回退到开发语言中文。
- 应用进入前台或收到系统 locale 变化通知后重新解析语言。
- 底部五个导航标签及无障碍标签改为直接使用 `LocalizedStringKey`，避免预先生成的普通 `String` 在运行时切换语言后保留旧值。
- 使用共享 `BNBULanguageSettings` 管理语言，个人页切换后无需重启即可刷新根视图。
- 课程状态、运动类别、运动项目、通知类别、通知筛选、免测项目、申请状态及外观选项等固定动态文案已改为查本地化资源。
- 教师、管理员和服务器录入的课程名、教师名、备注、通知正文等业务数据继续保持原文。

### 3.2 权限与隐私

说明：本轮只完成本机合规记录；服务端同意记录、撤回、数据删除申请仍依赖 API。

## 4. 未通过 / 不确定项

### P0：阻止正式环境启动或核心业务闭环

1. **Release API 地址仍为占位域名**
   - 当前为 `https://configuration-required.invalid/api`。
   - Release 启动会因配置校验失败而停止。
   - 需要正式 HTTPS 域名、证书和环境配置，不能在 iOS 端猜测。

2. **运动会话没有服务端契约**
   - 开始、暂停、继续、结束和 GPS 目前只保存在本机。
   - 现有提交未完整上传开始时间、结束时间、真实时长、暂停段、位置和运动日期。
   - 无法保证跨设备只有一个活动会话，也无法由服务端兜底每日一次、跨日和时间窗规则。

3. **正式测试条件缺失**
   - 缺少稳定测试账号、可重复测试数据和新运动流程的冻结 OpenAPI。
   - 尚未完成正式签名、Archive、安装及生产证书验证。

### P1：必须由 API 与 iOS 协作

1. 打卡开放时段固定为 06:00–22:00，未接全局、课程级、学期、节假日及特殊禁用日规则。
2. 学时仍固定为 20/10/10，未接规则版本、课程覆盖及动态目标。
3. 组织抵扣仍沿用“只能抵扣其他运动”的旧口径，v4 的动态分配尚未接入。
4. 成绩四项和权重仍硬编码，模型不能可靠区分未录入、0 分和缺考。
5. 成绩公示状态、历史版本、申诉、缓考、补考及归档结果未接入。
6. 校队和社团认证只有读取展示，没有新申请、证明上传、补充材料、教师审核和抵扣明细接口。
7. 忘记密码、新生设密、登录后改密和数据删除申请缺少接口。
8. 隐私同意时间及政策版本目前仅保存在本机，尚未上报服务端。
9. App 语言偏好未同步服务端，邮件和服务端通知无法按学生偏好生成。
10. FAQ 在线更新、服务反馈和工单系统无接口。
11. APNs 注册、device token 上传、服务端推送和通知去重未接入。
12. Push 只能跳到顶级 Tab，无法可靠定位到具体课程、申请、成绩异议或记录详情。
13. 后台被系统终止或更换设备后的活动运动恢复没有服务端支持。

### P1：客户端仍需继续修复

1. **全 UI 尚未完全双语**
   - 本轮已覆盖主导航和主要枚举。
   - `AppState`、校验规则及 Repository 中仍有大量客户端生成的中文错误字符串。
   - 部分带变量的上传进度、时长单位、无障碍说明和组合提示仍可能显示中文。
   - 服务器返回的任意原文必须与客户端固定错误分流，不能统一强制翻译。

2. **退出登录与草稿规则存在冲突**
   - v4 写明登出仅清 Token。
   - 当前登出和 401 会清理草稿、运动会话、凭证及待提交操作。
   - 直接保留会有不同账号读取旧账号本地数据的风险；应先完成账号隔离存储设计后修改。

3. **草稿自动保存不完整**
   - 当前主要依赖手动“保存草稿”。
   - 离开页面、进入后台和服务失败时未统一自动保存表单及所选凭证。

4. **打卡开始页信息不足**
   - 缺少更明确的今日状态、课程教师、课程/其他/总学时进度。
   - 部分数据需要 API，已有数据可先补 UI。

5. **免测本地规则不完整**
   - 默认项目未按学生性别选择 800m 或 1000m。
   - 证明校验仍允许仅视频，不符合“至少一张证明图片”的口径。
   - 最多两次提交需要服务端计数兜底。

6. **降级提示不完整**
   - 成绩缓存未统一显示“数据可能不是最新”。
   - 网络状态、服务维护状态和首次失败自动重试没有统一 UI。

7. **字符串目录仍有 94 个 stale 标记**
   - 其中既包含动态使用的有效键，也可能包含已删除调试入口。
   - 需要在完整页面双语回归后清理，避免误删仍在运行时查找的键。

### P2：测试、可访问性与发布

1. 真机需要验证：
   - 系统中文、系统英文及“跟随系统”；
   - App 内切换中文、英文后已打开页面立即刷新；
   - 相机、麦克风、定位及通知的允许、拒绝、受限和重新授权；
   - 前后台切换、锁屏、设备重启及跨日运动。
2. 英文自动化仍需扩展到所有正文、弹窗、帮助、隐私、成绩、申请及无障碍文案。
3. 完整本地 UI 套件已运行；3 个真实后端用例因缺少测试账号、密码和期望备注而跳过，仍需补跑。
4. `testSubmitDraftAndSubmittedRecordFlow` 虽通过，但出现一次 `Invalid frame dimension (negative or non-finite)` SwiftUI 运行时警告，需要定位具体布局表达式。
5. 需要执行 Release、Archive、正式签名及安装验证。
6. 需要检查 Dynamic Type、大字号、VoiceOver 和英文长文本布局。

## 5. 文档口径冲突与本轮决策

1. v4 要求首次按系统语言并支持中英文；产品另行确认“新手指引默认中文”。
   - 本轮决策：主应用默认跟随系统，新手引导固定简体中文。
2. v4 写明登出仅清 Token，但当前本地缓存未完全按账号隔离。
   - 本轮未直接保留草稿，避免跨账号数据泄露；列为后续架构修复。
3. 文档描述了密码重置、组织申请、成绩公示和历史成绩，但当前 API 不具备。
   - 本轮只在引导和帮助中明确标注尚未接入，不伪造可用入口。
